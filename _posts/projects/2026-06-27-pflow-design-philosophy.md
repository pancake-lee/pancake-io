如果你同时使用多个 AI 编程 Agent，比如 多个 Claude Code 实例同时在使用，还有 Hermes 在写文档，你很可能经历过这种时刻：

> Claude 处理完了在等你确认，但你切到别的窗口忙了一会儿，回来忘了它在等。只能逐屏翻找 tmux 会话，一个个 attach 进去看状态。

这个问题说大不大，说小不小。它不是技术难题，而是一种**微妙的注意力消耗**，你在多个 Agent 之间切换时，有一小部分心智始终悬在"那边怎么样了"的焦虑上。

PFlow 就是为这个场景设计的。但它的定位不是"又一个 AI Agent"，这篇文章想讲的，是它背后的设计哲学。

## 不是 Agent，是指挥沙盘

PFlow 不做对话，只做调度。

这个定位来自一个核心观察：**市面上不缺 Agent，缺的是在多个 Agent 之间管理注意力的工具。** 就像一位将军不需要亲自开枪，他需要一张沙盘，主力在哪，支援在哪，一眼看清哪路兵在打、哪路兵在等、该去哪路支援。

```text
用户设定主线/支线项目策略
       ↓
各路 Agent 在各项目中出战
       ↓
Dashboard 按项目聚合展示状态
       ↓
用户看 Dashboard → 判断：
  ├── 主线项目有 Agent 在等待 → 切过去处理
  ├── 支线项目有进展 → 快速浏览后继续主线
  └── 调整策略（升降级项目优先级）
       ↓
处理完毕 → 回到 Dashboard 或继续工作
```

**PFlow 是信息层，不替代用户已有的工作软件。** 用户在 Dashboard 获取信息后，自己决定何时切换到终端或 VSCode。PFlow 不做自动跳转，不接管编辑器，不替用户做决定。

这个"边界感"是整个产品最根本的设计原则。

## 从游戏设计到工作流：三幕认知引导

PFlow 的另一个设计源头，来自一次关于**游戏设计与工作效率**的深入讨论。

讨论的起点是一个有意思的观察：好的主机游戏能在几分钟内让你进入心流。你清楚地知道"要去哪"，但怎么去完全由你决定。而知识工作，哪怕是感兴趣的项目，往往需要一小时才能真正进入状态。为什么？关卡设计师到底做对了什么？

答案涉及到几个认知科学概念：

- **心流通道**：挑战与技能的动态匹配。太简单会无聊，太难会焦虑，中间那条窄道才是心流。好的关卡设计通过控制难度曲线（敌人强度、资源投放、谜题复杂度），让玩家始终待在这条窄道里。
- **宽线性结构**：只有一个明确的关卡出口（目标边界），但达成目标的具体路径保留充分自由度。《生化危机》的警察局关卡就是典型：你知道最终要逃出去，但先探索东翼还是西翼、先拿霰弹枪还是先解保险箱，是你自己的选择。
- **仪式心理学**：固定的动作序列能"欺骗"基底神经节，自动触发模式切换。就像游戏加载画面不只是掩盖加载时间，它在告诉你的大脑"现实世界暂停，游戏世界开始"。

> 关于两种游戏设计哲学的详细对比，以及它们如何迁移到知识工作场景，见[《游戏设计中的心流》](/pancake-io/posts/game-design-flow/)。

这套理论最终落地为 PFlow 的**三幕认知引导流程**：

### 第一幕：启动热身（认知卸载）

每天开始工作前，一般我们会先处理一下琐事，大致计划一下今天的工作，所以设计一个“启动页面”。

显示今日日期，一行柔和引导文案，底部一条缓慢填充的时间感知标尺。页面有一个输入框："今天只有一件事："。

用户在 30 分钟内自由处理邮件、通讯软件、琐事。唯一需要完成的动作是：**填入今日唯一的硬边界目标**，一句话，口头化、甚至情绪化都可以（"快点做，老板今天要验收"完全合法）。

进度条走完后只做微弱的背景色温变化，无任何警报，PFlow会一直等待你进入状态。

> **设计意图**：认知卸载（清空工作记忆缓存）+ 目标锚定。进度条替代倒计时，避免"损失厌恶"焦虑。

### 第二幕：切换仪式（模式锚定）

这是一个小小的仪式感，仪式内容完全由用户配置，让我们的大脑明确地切换一个工作模式。

全屏显示三个完成时态的复选框（可配置）：

```text
☐ 我已倒好一杯水/咖啡，放在手边
☐ 我已打开主线任务所需的全部窗口
☐ 我已关闭微信、企微及其他无关应用
```

用户必须**依次手动点击**三个复选框。全部打勾后确认按钮亮起。点击时播放极微弱的"咔哒"反馈动画。无时间限制，无催促。

> **设计意图**：手指点击复选框的动作，配合完成时态的陈述句，让本体感受参与"模式切换"宣告。这是软硬边界之间的保险栓，防止碎片化惯性污染深度工作时间。

### 第三幕：核心战局（心流通道）

前两幕的目的是帮你"进入状态"。第三幕则是**让你留在状态里**——它才是 PFlow 的核心。

#### 宽线性结构的 Dashboard 映射

回到前面讨论的宽线性结构：**一个明确的关卡出口 + 灵活的路径选择权。** 在 PFlow 中，这个结构被映射为 Dashboard 的项目优先级体系：

- **1 个主线项目** = 今日的关卡出口。这是你早上在第一幕写下的那句"今天只有一件事"。它不需要是精确的任务描述——"搞定 pflow 的提醒算法"和"快点做老板要的"都合法。关键不在于措辞，而在于**你确认了今天最重要的那个方向**。
- **最多 2 个支线项目** = 可选路径。当主线在等待（比如 Claude 正在跑一个长任务），你可以切到支线推进。但支线不会变成主线——系统会用提醒分数做反向牵引。
- **其余为普通项目** = 默认收起，不占据注意力空间。

1+2 的硬上限不是技术限制，而是**注意力保护机制**。同时追踪超过 3 个方向时，你在 Dashboard 上花的时间比在实际工作上还多——那就本末倒置了。

#### 红绿灯：余光中的态势感知

每个 Agent 会话在 Dashboard 上以一个状态指示灯展示：

- 🟢 **交战中**：Agent 正在执行任务，你不需要管它
- 🟡 **待命**：Agent 在等你——通常是需要授权工具调用或确认某个决策
- ⚪ **休整**：空闲，无活跃任务

这三种状态只需要**瞟一眼**就能分辨。不需要逐屏翻 tmux 会话，不需要 attach 进去看日志。就像开车时你不需要盯着仪表盘——余光扫到油表快见底了，你自然会找加油站。

#### 知觉信号：为何不用弹窗

状态展示解决了"看到"的问题，但还有一个更微妙的问题：**什么时候提醒你去看？**

最直觉的方案是弹窗——"支线项目有 Agent 已等你 5 分钟！"——但这恰恰是最糟的做法。

弹窗是**认知指令**：它抢占注意力焦点，强制中断当前思维流。被弹窗打断后，需要约 15 分钟才能恢复深度。这个数字不是感觉——中断代价理论（Disruption Cost Theory）有大量实证研究支撑。

PFlow 的选择是**知觉信号**：在项目卡片上叠加一层 CSS 遮罩，透明度随"提醒分数"缓慢变化。它用周边视觉传递信息，而非中央视觉抢占焦点。就像窗外天色渐暗——你知道该考虑开灯了，但它不会打断你正在做的事。

提醒分数综合四个维度动态计算：等待时长、专注持续、今日累计投入、项目优先级。核心行为只有三条：刚切换时 15 分钟不打扰、专注越久提醒越强（幂函数加速）、支线超时自动拉回主线。

#### 军情哨：建议，不是命令

Dashboard 右上角有一个"军情哨"区域——系统基于当前状态自动生成分析建议。

它不是 LLM 闲聊，而是一个基于规则的场景匹配引擎。当前覆盖约 20 个场景，分为紧急（主线 waiting > 5min）、关注（Agent 可能卡住了）、矫正（支线用时超过主线）、正面（一切正常）等类别。

但最关键的设计原则只有一条：**系统只做军情分析，不自动调兵。** 它告诉你"支线有个 Agent 已经等你 8 分钟了"，但它不会替你切过去。决定"切不切"的是你自己。

这是 PFlow 和那些"智能调度系统"最根本的区别。它不相信算法能替你做注意力决策——它只相信**给你足够好的信息，你比任何算法都清楚自己该做什么。**

#### 专注模式：玩家主动选择的深度投入

提醒系统是自动的，但有时候你比算法更清楚自己的状态。你知道接下来一小时绝对不能被打断。

专注模式就是这个场景的答案：点一下项目卡片上的"专注"按钮，系统为该项目设 15 分钟保护期。可连续叠加。保护期内非关注项目全部静默，只显示当前项目的状态变化。

到期自动解除，不做"永久免打扰"。这个设计是刻意的——**注意力保护应该是有意识的主动选择，而不是一次设置然后遗忘的配置项。** 就像游戏里你主动喝下一瓶专注药水，你知道它在 timer 内有效，过期了就该重新评估战场。

> Dashboard 的完整功能（路径即项目、提醒分数公式、遮罩层 CSS 实现、军情哨 ~20 场景详情、专注模式交互细节、换肤系统设想）见[《PFlow 产品全貌》](/pancake-io/posts/pflow-product-overview/)。

## 明确“不做”

### 主操作不是 Web 终端

Dashboard 提供了 Web 终端（基于 ttyd + tmux）作为辅助交互路径，但它不是主要入口，它不接管 Agent，用户可以直接使用原生的 Claude 等 CLI。
而使用`pflow claude`，用户可以在 Web 上直接打开终端，设计初衷是让用户在 Web 进行快速的授权操作，也并不建议作为主要交互路径。

总的来说，设计假设是：**用户在自己的终端里工作最舒适。** PFlow 不做"大而全的工作台"，而是"轻量级信息面板"。

### 接管待办列表

PFlow 的目标输入框只收一句话自由文本，不加"目标："等前缀标签。它不试图替代 Todoist、Jira 或用户的脑内清单。**工具应该做好自己的边界，而不是无限吞并其他工具的职责。**

## 技术实现

上面的三幕流程和设计决策，最终落地为一套完整的产品功能：Dashboard 两级视图、路径即项目机制、提醒分数算法、注意力遮罩、军情哨建议引擎、专注模式等。

技术架构和功能细节见[《PFlow 产品全貌》](/pancake-io/posts/pflow-product-overview/)，这里不再展开。

## 最后

PFlow 的"欢迎-仪式-战局"三幕流程，本质上是一段**认知开机引导程序**，它不帮你做事，它帮你在做事之前，把房间里的灯调对。

做这个项目的过程中，我反复确认一个问题：**"这个功能到底是在帮用户，还是在满足我自己作为开发者的存在感？"** 被砍掉的那些功能，大多是因为它们属于后者。

这也是 [程序员陷阱](/pancake-io/posts/programmer-trap/) 那篇文章的核心命题，**在 AI 让"实现"变得廉价的今天，最难的不是做出更多东西，而是做出对的东西。**
